## 当前代码优先修正

- `admin-api` 和 `app-api` 基础接口授权走 Better Auth session + `RbacGuard`；重复 RBAC runtime/graph engine 在 `libs/rbac-core`，菜单和权限读取各自 RBAC effective 权限与可见菜单。
- 公开接口统一使用 `@app/common` 的 `Public`，它只让 Better Auth session guard 不要求 session；读取会话仍使用 Better Auth 的 `Session`。
- `user_manager`、`role_manager`、`menu_manager`、`user_group_manager`、`task_manager` 均有 `permission list = view`。
- 权限管理页矩阵只维护核心 manager relation；对象例外授权通过 `object-bindings` 独立接口处理，属于后端/运维能力，前端可视化入口待迭代，不能写成菜单入口权限。
- 角色、用户、用户组、菜单关系表的筛选和分页走后端查询接口。
- 关系初始化接口只返回当前对象和 ID 集。
- system 管理页头部直接使用 Arco `a-page-header`。
- admin 基础授权由 RBAC 负责；SpiceDB 用于显式关系授权和关系数据页能力。
- 菜单只声明 `requiredPermissionCode`，角色授予权限，导航由 effective 权限匹配菜单声明生成。
- Promtail 当前只提升低基数字段 `app/env/log_type/level/module`，不提升 `user_id`，避免 Loki active streams 过高。

## 当前口径

- 认证使用 Better Auth session。
- 基础权限使用 RBAC 源表、effective 读模型和 `libs/rbac-core`。
- 系统关系表使用后端分页查询接口。
- 系统页头部使用 Arco `a-page-header`。

## 后续维护建议

- 规则发生变化时，先核对当前代码，再同步本目录。
- 涉及 admin 基础权限时，优先核对 RBAC Prisma schema、Service、前端页面和 `docs/admin-permission-system-code-architecture.md`；涉及 SpiceDB 数据页或对象关系授权时再核对 `spicedb/schema.zed`。
- 规则输入以当前代码和本目录 `active/` 文档为准。
