# Gene: Permission Change

## When

后台 RBAC 权限、菜单、角色、用户、用户组、任务、SpiceDB 数据页、按钮权限改动。

## Signals

- 文件路径包含 `rbac/`、`spicedb/`、`authz-permission`、`spicedb-data`、`role`、`menu`、`user-group`、`user`、`task`。
- 出现 `RbacAuthorizationService`、`SystemRbacGraphService`、`@RbacPermission`、`viewerCan`、`requiredPermissionCode`、`get_*_relations`、`query_relation_`、`query_assignable_`、`visibleMenuIds`、`visibleUserIds`。

## Instruction

1. 先核对当前代码、RBAC Prisma schema 和 `docs/admin-permission-system-code-architecture.md`。
2. Better Auth 只负责认证/session，RBAC 负责 admin 基础授权，SpiceDB 只负责显式关系授权。
3. 菜单只声明 `requiredPermissionCode`，角色授予权限，导航由 effective 权限匹配菜单声明生成。
4. 核心动作由 RBAC code 和 Service assert 控制。
5. Service assert 是最终安全边界。
6. 核心页面按钮只读接口返回的 `viewerCan...` 和 `meta.viewerCan...`。
7. 使用当前 RBAC 源表、effective 读模型和 SpiceDB 显式关系授权边界指导实现。
8. 改权限 code 或角色授权时，同步 DTO、Service assert、列表能力字段、前端按钮和文档。
9. 改权限关系后检查 RBAC effective rebuild 和用户状态版本刷新。
10. 系统关系抽屉使用后端分页时，要同步 `get_*_relations` 返回结构、DTO、Controller `query_*` 接口、Service 查询逻辑和前端 `GiTable :request`。
11. 关系初始化 DTO 只保留当前对象、能力字段和选中 ID 集；不要保留 `directUsers`、`assignableUsers`、`visibleUsers` 等完整候选数组字段。
12. 权限系统页头部直接使用 Arco `<a-page-header>`，不要把相关页面重新包装回 `NaPageHeader`。
13. 权限管理矩阵只维护核心 manager relation；对象例外授权走 `object-bindings` 独立接口，当前属于后端/运维能力，前端可视化入口待迭代，不要塞进菜单入口权限。
14. RBAC effective 表是读模型，源表才是授权配置事实；SpiceDB 投影表不参与 admin 基础授权。
15. 不要把 `admin-api` 的 SpiceDB 授权口径泛化到 `app-api`。
