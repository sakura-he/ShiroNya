# admin-web 前端与权限 UI 提示词

适用范围：`apps/admin-web`，尤其系统管理、权限、关系管理、Arco UI 页面。

## Prompt

你在 `apps/admin-web` 修改前端时，优先遵守当前 Arco Design Vue 和权限 UI 规则。核心权限按钮读取接口返回的能力字段，不要从菜单 requiredPermissionCode 自行推导动作权限。

## 页面头部规则

- system 管理页、权限页、任务页和用户详情页的标题栏直接使用 Arco 原生 `<a-page-header>`。
- 页面标题、副标题、返回行为和页头操作统一放在 `<a-page-header>` 的 `title`、`subtitle`、`extra` 插槽里表达。
- 不要新增只包裹 `<a-page-header>` 的同类壳组件。

## Arco Card 规则

- 页面级卡片、表格外壳、关系面板、空状态外壳、图谱画布外壳等需要卡片视觉时，优先使用 Arco 原生 `<a-card>`。
- 避免用 `div`、`section` 或自定义组件手写 card 效果。
- 避免新增 `relation-block`、`surface`、`dict-surface`、`dict-empty`、`task-table-shell` 这类模拟 card 外观的样式类。
- 如果是布局容器、表单栅格、代码块、图谱节点、VueFlow/G6 内部节点或领域专用可视化元素，可以保留自定义样式。
- 不要在页面里局部写死 `.arco-card-header` 高度。

## 表单与表格规则

- 新增或重构前端表单时，优先使用 `@form-create/arco-design` 和 `<form-create>`，保持与系统页现有搜索表单、抽屉表单、内嵌关系筛选表单一致。
- 表单规则、校验、字段显隐和组件属性优先用 form-create 的 rule/options/API 表达，不要另起一套手写表单状态管理。
- 列表、分页、关系明细、抽屉内局部表格等表格场景优先使用项目内 `GiTable` 组件。
- 表格列定义、分页参数、刷新动作和后端请求适配优先沿用 `GiTable`、`ProColumns`、`GiTableRequestParams`、`RequestData` 等现有类型。
- `GiTable` 工具栏操作按钮使用 `#custom-extra` 插槽；不要把 `GiTable` 顶部操作写进 Arco Table 的普通 `#extra` 语义里。
- 不要在系统页为常规数据表新写一套 `a-table` 外壳；只有 `GiTable` 无法覆盖的特殊展示，再说明原因并保持 Arco 视觉一致。

## form-create 状态与弹窗规则

- 不要把 `@form-create/arco-design` 的 `$f.clearValidateState()` 当成中性“清校验状态”使用；它会把 Arco Form 字段写成 `success` 状态，导致空输入框出现绿色成功态边框。
- 需要清掉打开弹窗后的上一次校验状态时，在当前页面直接通过 `$f.formEl()` 拿到底层 Arco Form 实例调用 `clearValidate()`；不要为了这件事新增公共封装工具、公共 util 或通用抽象层。
- 删除、软删除、禁用、重置密码等确认弹窗，每次点击操作都先重置对应表单数据、表单 API 值、校验状态、目标对象和操作模式；确认按钮里先复制当前 `targetId`，再 `await validate()` 和请求接口，避免连续操作读到上一次目标。
- form-create 自定义控件保持受控组件语义：只传 `modelValue` 与 `onUpdate:modelValue`，不要同时传会随输入变化的动态 `defaultValue`，避免输入一个字母后内部状态重置、光标跳动或失焦。
- 编辑弹窗如果表格行已经包含可编辑字段，直接用当前行快照填充表单并对比变更；不要为了打开编辑弹窗再次请求详情接口。详情抽屉、只读详情或缺字段场景才请求详情。
- 搜索表单尽量覆盖主要可筛选列，例如 ID、姓名、昵称、邮箱、手机号、状态、角色、验证状态、创建时间、更新时间等，并由 `GiTable :request` 走后端分页筛选。
- 排查 form-create 绿色空态或焦点异常时，先在当前页面直接搜索 `clearValidateState`、动态 `defaultValue`、弹窗 target/form reset 顺序，再做最小修复。

## 关系管理页面规则

- 角色、用户组、菜单等系统关系管理区域优先使用 `<a-card>` 承载。
- 关系表达式放在 Card 的 `#title` 插槽中，用 Arco `<a-typography-text code>` 展示。
- Card 右上角的保存、授权、分配等操作按钮放在 Card 的 `#extra` 插槽内；如果这些按钮属于 `GiTable` 顶部工具栏，则放进 `GiTable #custom-extra`。
- 不要手写 Card header、hint、relationship tag 的卡片视觉。
- 主列表下方不要默认添加展开行摘要展示关系焦点、计数或路径；关系详情由操作列打开抽屉承载，除非用户明确要求展开行。
- 系统关系表的筛选和分页走后端接口，不在浏览器侧对完整候选数组自行处理筛选或页码。
- 不要在前端持有完整候选集后自行分页/筛选。
- 关系初始化接口只承载当前选择 ID、能力字段和抽屉初始状态；实际表格行通过 `GiTable :request` 按页请求后端。
- 角色抽屉初始化 DTO 只保留 `role`、`directUserIds`、`userGroupIds`、`effectiveUserIds`。
- 用户抽屉初始化 DTO 只保留 `user`、`userGroupIds`、`visibleMenuIds`、`inheritedRoleIds`；`user` 自身保留 `roleIds` / `effectiveRoleIds`。
- 用户组抽屉初始化 DTO 只保留 `group`、`memberUserIds`、`roleIds`、`visibleMenuIds`。
- 菜单抽屉初始化 DTO 只保留 `menu`、`roleIds`、`visibleUserIds`。
- 初始化 DTO 不包含 `directUsers`、`assignableUsers`、`memberUsers`、`roles`、`visibleMenus`、`visibleUsers` 这类完整候选数组。
- 角色、用户、用户组、菜单关系抽屉当前使用的后端分页接口包括：
    - `POST /role/query_assignable_users`
    - `POST /role/query_assignable_user_groups`
    - `POST /role/query_effective_users`
    - `POST /user/query_relation_roles`
    - `POST /user/query_relation_user_groups`
    - `POST /user/query_relation_menus`
    - `POST /user/query_user_sessions`
    - `POST /user-group/query_relation_members`
    - `POST /user-group/query_relation_roles`
    - `POST /user-group/query_relation_menus`
    - `POST /menu/query_relation_roles`
    - `POST /menu/query_visible_users`
- 草稿选择态通过 `draftUserIds`、`draftUserGroupIds`、`draftRoleIds` 传给后端，后端返回当前页行的选中/禁用状态。

## 抽屉加载顺序

点击“编辑关系”“查看关系”等需要远程加载数据的抽屉时：

1. 清空上一次关系数据或设置当前 ID。
2. `drawerVisible = true`。
3. `await loadRelations(...)`。

不要先 `await loadRelations(...)` 再打开抽屉。网络慢时会造成点击无反馈，也容易显示上一次数据。

## 权限按钮规则

核心用户、角色、菜单、用户组、任务页面：

- 创建按钮读取 `meta.viewerCanCreateX`。
- 用户页创建按钮看 `userMeta.viewerCanCreateUser`，行按钮看 `viewerCanUpdate/Delete/ResetPassword/ViewSession/RevokeSession`。
- 角色页创建按钮看 `roleMeta.viewerCanCreateRole`，行按钮看 `viewerCanUpdate/Delete/AssignUser/AssignUserGroup/AssignTaskCapability/AssignTaskResource`。
- 菜单页创建按钮看 `menuMeta.viewerCanCreateMenu`，详情操作看 `selectedMenu.viewerCanUpdate/Delete/AssignRole`。
- 用户组页创建按钮看 `groupMeta.viewerCanCreateUserGroup`，成员和角色分配看当前用户组对象的 `viewerCanAssignMember/AssignRole`。
- 任务页创建按钮看 `taskMeta.viewerCanCreateTask`，行按钮看 `viewerCanUpdate/Delete/Run/ViewLog`。
- 不要用菜单 permission 判断核心动作按钮。

示例口径：

```vue
<a-button v-if="userMeta.viewerCanCreateUser">创建用户</a-button>
<a-link v-if="record.viewerCanUpdate">编辑</a-link>
<a-link v-if="record.viewerCanDelete">删除</a-link>
<a-button v-if="taskMeta.viewerCanCreateTask">创建任务</a-button>
<a-link v-if="record.viewerCanRun">运行</a-link>
```

仍可使用 `hasPermission()` 或 `<ShiroAuth>` 的场景：

- 页面入口和隐藏页路由守卫，例如 `system.user.view`、`system.user.detail`。
- 字典和 SpiceDB 数据页面这类非核心 manager 按钮。

## 权限管理页规则

- 权限管理页菜单入口是 `system.permission.view`，前端页面是 `apps/admin-web/src/views/system/permission/Permission.vue`。
- `GET /authz-permission/matrix` 返回 SpiceDB schema definition、核心 manager 模块、角色列表、当前授权状态和展示名。
- `POST /authz-permission/matrix/preview` 预览批量变更的 tuple 增量、影响角色和影响用户。
- `POST /authz-permission/matrix/apply` 应用批量变更；当 preview 返回需要确认时，前端必须传 `confirmedLargeChange: true`。
- `POST /authz-permission/rename-resource` 按 SpiceDB 实体名更新页面展示名。
- 页面提供“按实体”和“按角色”两个视图；单 relation 保存、模块保存和全页保存都会汇总为批量 `changes`，先预览再应用。
- “按实体”左侧和详情标题只读取 `authz_resource_metadata.display_name`；缺失元数据行由 `matrix` 接口按英文实体名补齐，不在前端写死中文映射。
- 可写范围只包含 `user_manager`、`role_manager`、`menu_manager`、`user_group_manager`、`task_manager`。
- 非核心 definition 的 relation/permission 只作为 schema 探测结果展示，不提供授权控件。
- 角色没有对应编辑能力时，选择项和 checkbox 禁用。
- 菜单 Page/Catalog 入口仍在菜单页维护，不在权限管理页处理。

对象例外授权：

- 当前代码存在 `object-bindings`、`object-bindings/preview`、`object-bindings/apply` 接口。
- 对象例外授权仍使用权限管理页入口 `system.permission.view`，但不是菜单入口权限，不要把对象例外写成菜单 permission。
- 大批量对象例外授权 apply 需要按 preview 结果传 `confirmedLargeChange`。
- 当前对象例外授权属于后端/运维能力，权限管理页的可视化产品入口待后续迭代；不要把现状描述成前端已经完整覆盖对象例外授权。

## 菜单与 Permission Token

- 页面入口和隐藏页由 `/account/navigation.permissions` 中的 RBAC code 控制。
- `/account/navigation` 的 `menus` 排除 Button，用于左侧导航和动态路由。
- `/account/navigation.permissions` 返回当前用户拥有的 RBAC 权限 code，适合入口、隐藏页和非核心 manager 按钮。
- `rbac_menu.requiredPermissionCode` 使用 `<scope>.<resource>.<action>` 三段式，全 ASCII 小写 + 数字 + kebab。
- 菜单只声明“所需权限”，不使用菜单-权限绑定 UI。
- 权限分组只用于管理页面组织数据；权限和菜单各自通过 `groupId` 归属到单个分组，不参与运行时判权。
- RBAC 权限页可以维护分组以及分组下的权限/菜单；RBAC 角色页的“按角色授权”按分组展示权限，保存的是角色-权限关系。

| 菜单类型    | action 规则                     |
| ----------- | ------------------------------- |
| Catalog     | `menu`                          |
| Page 主页面 | `view`                          |
| Page 隐藏页 | `detail`                        |
| Button      | 仅用于非核心 manager 的模块按钮 |

## 状态版本刷新

- `apps/admin-web/src/api/index.ts` 会保存响应里的 `x-user-state-version`。
- 请求拦截器会带上本地版本头 `x-user-state-version`。
- 普通业务响应带 `x-user-state-changed: 1` 时，触发一次去重后的 `refreshAuthStateByVersionChange()`。
- `/account/navigation` 是刷新链内部调用，它的响应只保存版本头，不触发自身刷新。
- `refreshAuthStateByVersionChange()` 负责串行刷新 Better Auth session、权限数组和动态菜单。

## 验证命令

修改系统页面卡片或关系 UI 后至少运行：

```powershell
pnpm --filter ./apps/admin-web exec prettier --write <changed files>
pnpm --filter ./apps/admin-web exec vue-tsc --noEmit
```

涉及图谱组件、全局样式或多个页面结构时继续运行：

```powershell
pnpm --filter ./apps/admin-web build
```

提交前扫描系统页面，确认没有重新引入手写 card 外壳类或关系展开摘要。
