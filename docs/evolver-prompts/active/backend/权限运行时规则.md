# RBAC 权限运行时提示词

适用范围：`admin-api` 和 `app-api` 基础 RBAC 权限、菜单、角色、用户、用户组、任务、RBAC 管理页、SpiceDB 数据页和显式关系授权。

## Prompt

你在 ShiroAdmin 中处理基础权限时，以当前代码、`prisma/admin/rbac.prisma`、`prisma/app/authz.prisma`、`apps/*/src/modules/system/rbac/**`、`libs/rbac-core/src/**`、核心 Service、前端页面和 `docs/admin-app-permission-authz-guide.md` 为准。Better Auth 负责认证和 session；RBAC 负责 admin/app 基础授权；SpiceDB 只负责显式关系授权功能和关系数据运维。

## 核心结论

```text
基础接口判权 = RBAC permission code
角色授予权限 = rbac_role_permission
菜单声明所需权限 = rbac_menu.required_permission_code
导航可见性 = 用户 effective permission codes 匹配菜单 requiredPermissionCode
Service assert 是最终安全边界
```

RBAC 数据缺失时应显式报错或明确拒绝。

## 后端规则

- 全局 Guard 链是 `BetterAuthSessionGuard` + `RbacGuard`。
- `RbacGuard` 是应用适配器，实际共享逻辑在 `libs/rbac-core/src/runtime/rbac-guard.ts` 的 `RbacGuardBase`。
- `@RbacPermission(code)` 语义是声明 RBAC 权限 code；它会写入权限发现候选 metadata，并通过 `RbacDeclarePermissions({ permissionCode: code })` 写入 `RbacGuard` 读取的运行时 metadata。
- `RbacAuthorizationService` 是应用适配器，实际共享判断顺序在 `RbacAuthorizationServiceBase`；应用 store 负责读取本应用数据库中的通用 `rbac_*` Prisma 表。
- `SystemRbacGraphService` 负责 Prisma 读写和 user-state，角色继承、用户组角色合并、权限集合和可见菜单集合的纯计算在 `libs/rbac-core/src/graph/rbac-graph.engine.ts`。
- code 不存在、禁用或软删除时按配置错误处理。
- `account` 当前用户自助接口、上传、Prometheus metrics 和显式 `@Public()` 接口不需要菜单权限点。

## RBAC 源表与读模型

源表：

admin/app 已拆成两个数据库，表名和 Prisma model 都使用同构通用命名。

- `rbac_role`
- `rbac_user_group`
- `rbac_permission`
- `rbac_menu`
- `rbac_user_role`
- `rbac_user_group_member`
- `rbac_user_group_role`
- `rbac_role_permission`

读模型：

- `rbac_effective_user_role`
- `rbac_effective_user_permission`
- `rbac_user_visible_menu`

基础授权使用 `rbac_menu.required_permission_code`，不使用菜单-权限多对多绑定。

## 菜单规则

- 菜单字段使用 `requiredPermissionCode` / `required_permission_code`。
- 菜单不授予权限，只声明可见所需权限。
- 角色页授予权限；菜单可见性由 effective 权限匹配产生。
- Catalog/Page 进入导航，Button 不进入 `menus`，但 Button code 可以进入 `permissions`。
- Page 只能挂在 Catalog 下，Button 只能挂在 Page 下，根节点只能创建 Catalog。

## 权限分组规则

- `rbac_permission_group` 只用于后台权限管理归类，不参与 `RbacGuard`、`SystemRbacGraphService` effective 展开或 `/account/navigation` 判权。
- `rbac_permission.group_id` 和 `rbac_menu.group_id` 都是单值归属；一个权限或菜单最多属于一个分组。
- 新增/编辑权限、菜单时可以写 `groupId`；权限页分组管理可以集中改写权限和菜单归属。
- 角色授权仍然写 `rbac_role_permission`，不要把分组当成角色授权来源。

## 关系表分页查询

系统关系管理页的表格筛选和分页由后端接口处理，前端通过 `GiTable :request` 请求当前页数据。关系初始化接口只返回当前关系 ID、能力字段和抽屉状态，不返回完整候选表格。

不要把 `directUsers`、`assignableUsers`、`memberUsers`、`roles`、`visibleMenus`、`visibleUsers` 这类完整候选数组重新塞回初始化 DTO。

当前分页查询接口包括：

- `POST /rbac/role/query_relation_users`
- `POST /rbac/role/query_relation_groups`
- `POST /rbac/role/query_relation_permissions`
- `POST /rbac/user/query_relation_roles`
- `POST /rbac/user/query_relation_groups`
- `POST /rbac/user/query_relation_menus`
- `POST /rbac/user-group/query_relation_members`
- `POST /rbac/user-group/query_relation_roles`
- `POST /rbac/user-group/query_relation_menus`
- `POST /menu/query_relation_roles`
- `POST /menu/query_visible_users`

草稿选择态通过 `draftUserIds`、`draftGroupIds`、`draftRoleIds`、`draftPermissionIds` 传给后端，后端返回当前页行的选中状态。

## SpiceDB 保留范围

SpiceDB 仍可用于：

- `system/spicedb-data` relationship、schema、permission check、explain、watch 管理。
- 任务或业务对象的显式关系授权。
- 运维调试和关系图解释。

admin 用户、角色、用户组、菜单、权限的基础授权管理由 RBAC 负责。

## 前端按钮字段

核心用户、角色、菜单、用户组、任务页面：

- 创建按钮读取 `meta.viewerCanCreateX`。
- 行按钮读取 `record.viewerCanUpdate/Delete/...`。
- 不要用菜单 permission 自行推导核心动作按钮。

仍可使用 `hasPermission()` 或 `<ShiroAuth>` 的场景：

- 路由入口和隐藏页守卫。
- 字典、SpiceDB 数据页这类非核心 manager 按钮。

## 状态版本刷新

- 用户、角色、用户组、角色权限、菜单 requiredPermissionCode 变化后，需要重建 RBAC effective 读模型。
- 影响导航时调用 `AdminUserStateService.bumpMenuStateVersion()`。
- 影响角色授权时 bump 相关角色版本。
- `/account/navigation` 缓存 key 包含综合状态版本，版本变化后自然失效。
