# Nest / Prisma / Controller / DTO 提示词

适用范围：后端 Controller、Service 返回、DTO、Swagger、Prisma 数据处理，以及 `app-api` 当前业务接口口径。

## Prompt

你在 ShiroAdmin 后端修改代码时，要保持 NestJS、Prisma、Zod DTO、Swagger 和当前响应包装约定一致。具体写法以当前模块现状为准。

## Controller 规则

- Controller 通常直接返回业务数据。
- 应用级 `ResponseFormatInterceptor` 会包装为 `{ data, code, message }`。
- 避免在 Controller 手动返回 `{ data: result }`，除非当前接口协议确实如此。
- 后端 HTTP 接口统一只允许 `GET` / `POST`，禁止 RESTful 资源风格、`PUT` / `PATCH` / `DELETE`、`/:id` 资源路径和同类变体。
- 路径命名统一采用 action-style snake*case，列表、筛选、搜索类 `POST` 接口统一用 `query*\*`，例如 `query_role_list`、`query_user_list`；写操作继续使用 `update_user_group`、`assign_permissions`。
- 一个 Controller 只负责一个明确子域。不要把 `role`、`user-group`、`permission`、`menu` 等子模块接口写进同一个聚合 Controller；应按子域拆分 controller/module，公共前缀可以放在各子域 Controller 的 `@Controller()` 路径里。
- Swagger 当前倾向使用 `@ApiOkResByZod({ summary, description, type: XxxDto })`。
- 如果模块已有稳定的 Swagger 写法，先保持局部一致，再考虑统一。
- 权限装饰器只在当前模块真实启用时沿用。

### 应用差异

`app-api`：

- 使用 Better Auth session。
- 当前账户接口常用 `@Session()` 读取 `AppSession`。
- 基础接口授权走 `BetterAuthSessionGuard` + `RbacGuard`，业务控制器优先用 `@RbacPermission()` 声明 RBAC permission code。

`admin-api`：

- 使用 Better Auth session。
- `AppModule` 注册全局 `BetterAuthSessionGuard` 和 `RbacGuard`。
- 管理端控制器常配合 `@RbacPermission()`、`RbacAuthorizationService` 和 service 层断言。
- 新增接口时先看相邻控制器如何传入 `actorId`，再沿用模块现有权限写法。

## DTO 规则

- 响应 DTO 通常保留纯业务 Schema，运行时响应包装交给拦截器。
- 请求 DTO 通常继承 `createZodDto(...)`，便于 `nestjs-zod` 管道验证。
- 能复用 `prisma-zod-generator` 生成 schema 时优先复用；复杂业务响应可以手写 Zod 组合。
- admin 侧部分 DTO 同时使用手写 Zod 和 Swagger `ApiProperty`，修改时先保持当前模块风格。
- 响应 schema 不手动包 `{ data: ... }`，除非该接口协议确实如此。
- 密码、salt、内部 token、密钥等敏感字段不出现在响应 DTO 中。
- 复杂 include 或权限矩阵可以先用明确的手写 schema；短期难以稳定时再用 `z.unknown()` / `z.any()` 并补说明。
- 文件命名以当前模块为准；常见形式是 `dto/req.dto.ts`、`dto/res.dto.ts` 或 `dto/{module}.dto.ts`。

## Prisma DateTime 规则

- `libs/prisma-admin/src/extended-client.ts` 与 `libs/prisma-app/src/extended-client.ts` 已启用 DateTime 转 ISO 字符串扩展。
- Service/Controller 中新增日期转换前先确认运行时类型。
- 不要对已经是字符串的 Prisma DateTime 查询结果调用 `.toISOString()`。
- `asPrismaISODate(...)` 更适合作为类型标注/边界辅助。

## App API 当前 API 口径

`app-api` 是用户端后端服务：

- 服务端口：3001。
- 全局 API 前缀：`/app`。
- Swagger 文档：`http://localhost:3001/api-docs`。
- 认证基础：Better Auth session，具体路由由 Better Auth 和插件配置挂载。

### Account

| 方法 | 路径                       | 说明                                        |
| ---- | -------------------------- | ------------------------------------------- |
| GET  | `/app/account/info`        | 当前登录用户、会话、角色、profile、权限快照 |
| GET  | `/app/account/menus`       | 当前用户可见菜单                            |
| GET  | `/app/account/permissions` | 当前用户动作权限映射                        |

当前 `app-api` 没有 `bind_phone`、`profile` 这类 Account Controller 方法。手机号、资料更新或认证能力应先查 Better Auth 插件和现有业务模块。

### 其它业务接口

- User：`query_user_list`、`create_user`、`update_user`、`delete_user`。
- Menu：`query_all_menus`、`detail`、`create_menu`、`update_menu`、`delete_menu`。
- Chat：当前存在 `chat/session`、`chat/message`、`character` 等控制器；新增或调用前先查看 `apps/app-api/src/modules/chat` 的实际 DTO 和路由。
- Dict、Upload、System Task、Audit Log、SAT 等接口以对应 controller 为准。

## App API 数据模型口径

当前用户端模型以 Better Auth 相关表为主：

- `AppBetterAuthUser`
- `AppBetterAuthSession`
- `AppBetterAuthAccount`
- `AppBetterAuthUserProfile`
- `AppBetterAuthApiKey`
- `AppBetterAuthUserRole`
- `AppRole`
- `AppMenu`

当前用户端模型以 Better Auth 和 RBAC 相关表为准。

## App API 权限现状

App 端基础菜单和动作权限走 RBAC：

- 菜单通过 `requiredPermissionCode` 声明可见性。
- 角色授予权限，effective 读模型计算用户最终权限。
- `/account/navigation`、`/account/menus`、`/account/permissions` 读取 effective 权限和可见菜单。

如果后续调整真实权限过滤，需要同步 `AccountService`、前端缓存策略和本提示词。

## 生成/修改代码检查清单

- Controller 是否直接返回业务值。
- 响应是否交给拦截器包装。
- Swagger DTO 是否排除了敏感字段。
- DTO 是否没有重复包 `{ data }`。
- 是否没有对 Prisma DateTime 字符串调用 `.toISOString()`。
- 当前接口是否需要公开、登录或 SpiceDB 能力控制。
- 会话信息是否能直接满足当前用户/角色读取需求。
