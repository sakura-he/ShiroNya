# 日志与可观测性提示词

适用范围：日志、审计、HTTP/gRPC 请求链路、Loki、Promtail、异常排查。

## Prompt

你在 ShiroAdmin 中新增或修改日志能力时，要复用当前统一日志系统，不要绕开已有中间件、拦截器、审计服务和运行时 logger。日志文档只记录当前日志链路。

## 当前日志入口

```typescript
import { createRuntimeLogger } from '@app/common';

const logger = createRuntimeLogger('my_module');

logger.info('完整结构日志', { id: 1 });
logger.info.title('仅标题行', { id: 1 });
logger.warn('警告信息');
logger.error('错误信息', { error });
```

`createRuntimeLogger(moduleName)` 返回：

- `info`
- `debug`
- `verbose`
- `warn`
- `error`
- `system`
- `userAction`
- `audit`

其中五个级别方法都带 `.title()` 变体。普通调用会输出标题行和开发环境 detail，`.title()` 只输出标题行，并将参数内联到标题行末尾。

## 运行时实现

- `libs/common/src/logger/runtime-logger.ts` 使用 Winston + `winston-daily-rotate-file`。
- 日志模块是纯函数入口，不依赖 Nest DI；在 bootstrap、模块配置和普通 service 中都可以使用。
- Console transport 保留 Nest 风格标题行。
- DailyRotateFile transport 按 app + logType 懒挂载。
- 颜色由 `SHIRO_LOG_COLORS` 统一定义，经 `winston.addColors()` 注册后使用。

## 当前日志分层

- `system`：系统运行、外部依赖、异常、后台任务、服务健康等。
- `user_action`：用户操作轨迹和普通业务行为。
- `audit`：登录、注册、改密、任务操作、菜单角色授权、权限拒绝等高风险留痕。

## 文件落盘与 Loki

- 审计日志 `logType === 'audit'` 只输出控制台，不落盘文件；审计日志继续走数据库，不接入 Loki。
- 其它日志写入 `{LOKI_LOG_DIR}/{appName}/{logType}-%DATE%.log`。
- 文件格式为 JSON Lines，每行是完整 `RuntimeLogEntry`。
- `LOKI_LOG_DIR` 未设置时使用 `logs/loki`。
- 生产 Debian 环境可设置 `LOKI_LOG_DIR=/var/log/shiro-nya/loki`。
- `SHIRO_APP_NAME` 由应用启动入口设置，未设置时为 `unknown-app`。

生产环境日志路径口径：

```text
/var/log/shiro-nya/loki/app-api/system-YYYY-MM-DD.log
/var/log/shiro-nya/loki/app-api/user_action-YYYY-MM-DD.log
/var/log/shiro-nya/loki/admin-api/system-YYYY-MM-DD.log
/var/log/shiro-nya/loki/admin-api/user_action-YYYY-MM-DD.log
```

本地开发日志路径口径：

```text
logs/loki/app-api/system-*.log
logs/loki/app-api/user_action-*.log
logs/loki/admin-api/system-*.log
logs/loki/admin-api/user_action-*.log
```

## Promtail 规则

- Debian 上使用 `docker/config/promtail/promtail-config.yaml` 作为 Promtail 主配置。
- 本地 Windows 推送远端 Loki 时使用 `docker/config/promtail/promtail-local.windows.yaml` 和 `scripts/start-local-loki.ps1`。
- 本地 Promtail positions 文件写到 `logs/promtail-local-positions.yaml`，关闭后下次从上次位置继续推送。
- 当前仓库 Promtail 配置只提升低基数字段为 Loki labels：`app`、`env`、`log_type`、`level`、`module`。
- 当前 Promtail 配置不提升 `user_id`，避免 `id/user_id` 打爆 Loki active streams；需要按用户筛选时应查询 JSON 内容，而不是把高基数字段提升为 label。
- 日志内容保留原始 JSON 行，便于在 Loki 中查看完整结构。

Loki 查询示例：

```logql
{job="shiro-nya-runtime-logs"}
{job="admin-api-local-runtime-logs"}
{app="app-api",log_type="system"}
{app="app-api",log_type="user_action"}
{app="admin-api",log_type="system"}
{app="admin-api",log_type="user_action"}
```

## 编码规则

- HTTP 请求日志由中间件和上下文拦截器处理，不要在每个 Controller 手写重复请求日志。
- 高风险写操作优先使用当前 `@AuditLog` 和审计上下文扩展能力。
- 权限拒绝场景如果已有审计上下文，应让审计链路记录拒绝结果。
- 日志中不要泄漏密码、token、cookie、secret、验证码、证书内容。
- 二进制、文件对象、大响应体要按当前标准化/脱敏策略处理。
- gRPC 异常日志需要保留 requestId 串联能力。
- 不要为了 Loki 检索便利把高基数字段新增为 labels；先确认 Promtail 配置和 stream 基数风险。

## 修改时检查清单

- 新增模块是否需要审计。
- 是否复用了当前 logger，而不是 `console.log`。
- 是否保留 requestId / userId / action / resource 等关键上下文字段。
- 是否不会在生产日志中输出敏感值。
- 新增日志级别时，是否同步 `SHIRO_LOG_LEVELS`、`SHIRO_LOG_COLORS` 和相关类型。
- 调整控制台格式时，是否确认文件 transport 不被 ANSI 颜色污染。
- 调整 `user_action` detail 裁剪时，是否不影响 JSON Lines 中的完整结构。
- 修改 Loki/Promtail 时，是否同步检查 `docker/config/promtail/promtail-config.yaml` 和 `docker/config/promtail/promtail-local.windows.yaml`。
- 是否需要更新模块文档中的日志或审计说明。
