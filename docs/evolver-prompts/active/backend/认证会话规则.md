# Better Auth Session 提示词

适用范围：认证、登录态、session、当前用户、Account 聚合、改密、重置密码、公开接口。

## Prompt

你在 ShiroAdmin 中处理认证时，以 Better Auth session 为基础。认证路由由 Better Auth 及其插件挂载；需要确认具体接口时，优先查看对应 `*-options.ts`、Better Auth service、OpenAPI 输出或运行时路由。

## 当前模块

| 应用        | 模块路径                                   | 说明                                                  |
| ----------- | ------------------------------------------ | ----------------------------------------------------- |
| app-api    | `apps/app-api/src/modules/better-auth`    | 用户端 Better Auth 配置、session 类型、用户资料扩展   |
| admin-api | `apps/admin-api/src/modules/better-auth` | 管理端 Better Auth 配置、管理员 session、用户管理 BFF |

## admin-api 当前口径

- `admin-api` 登录入口基于 Better Auth cookie session。
- 前端通过 HttpOnly Cookie 建立和恢复 `admin-api.session`。
- 未公开的后台接口以 Better Auth session 作为认证基础。
- `BetterAuthService` 通过 `rbac_effective_user_role -> rbac_role` 读取有效角色，再回 profile。
- `session.roles` 的角色 ID 来自 RBAC effective 读模型；session 不保存权限全集。
- `BetterAuthSession` 包含 `user`、`session`、`roles`、`profile`。
- 后台基础业务授权由 `RbacGuard`、`@RbacPermission()` 和 service 层 assert 负责，Better Auth 本身不承担核心业务授权。
- `@RbacPermission(permissionCode)` 是 RBAC 权限 code 入口声明，会复用 `RbacDeclarePermissions({ permissionCode })` 并写入 `RbacGuard` 读取的运行时 metadata。

## app-api 当前口径

- `app-api` 主 HTTP 入口使用 Better Auth session。
- Better Auth 插件包含 Expo、admin、API key、bearer、openAPI、phoneNumber、customSession 等能力。
- `AppSession` 包含 Better Auth 用户、会话、角色和 profile。
- `AppBetterAuthService` 负责短信 OTP 限流、secondaryStorage、新用户 profile 初始化、默认角色绑定和用户状态版本刷新。
- Account 模块负责“当前登录人”视角的数据聚合，优先复用 session 中的用户、角色和 profile，减少为了当前用户信息重复回表。
- `app-api` 基础接口授权同样走 Better Auth session + `RbacGuard`；菜单声明 `requiredPermissionCode`。

## Better Auth 关注点

- session 是后端业务读取当前用户、角色和 profile 的主要来源。
- Better Auth 负责“你是谁”和 session 生命周期。
- RBAC 或业务 Service 负责 admin 基础权限判断；SpiceDB 只用于显式关系授权功能。
- 用户名、手机号、OpenAPI、Expo、API Key 等能力以当前 Better Auth 插件配置为准。
- Swagger/OpenAPI 的安全说明可能仍写 bearer，但实际认证介质可能是 cookie、bearer 插件或客户端 SDK 组合；联调前应核对当前 Better Auth 配置。
- 改密、重置密码、会话撤销等能力优先复用 Better Auth 官方/admin API。

## 编码规则

- 公开接口统一使用 `@app/common` 导出的 `@Public()`；它只写 Better Auth session 公开标记。不要直接从 `@thallesp/nestjs-better-auth` 导入 `Public` 到业务 controller。
- 读取当前用户优先使用 `@thallesp/nestjs-better-auth` 的 `@Session()` 或 `request.session`。
- 已认证请求中的角色信息优先使用 session 内的角色数据，不要为了角色名重复查表。
- Account 响应中避免泄露密码、内部 token、secret、验证码等敏感字段。
- `account` 当前用户自助接口、上传、Prometheus metrics 和显式 `@Public()` 的接口不走菜单权限点。

## admin-api navigation

- `/account/navigation` 先读取 RBAC effective 可见菜单 ID，再回 `rbac_menu` 按 `status = ENABLE` 和菜单类型过滤。
- `menus` 排除 Button，用于左侧导航和动态路由。
- `permissions` 返回当前用户拥有的 RBAC permission code；Button code 会进入权限列表，但核心用户、角色、菜单、用户组、任务页面不从这里推导动作按钮。
- 隐藏详情页仍是 Page 菜单，例如 `system.user.detail`，由路由和页面入口 guard 控制。
- 菜单只声明 `requiredPermissionCode`，角色授予对应权限后用户才会看到菜单。

## 回归关注点

- 登录后 session 是否包含用户、profile、有效角色。
- `admin-api` 角色/用户组禁用后重新登录，session.roles 是否按 RBAC effective 与启用角色元数据更新。
- 公开接口是否确实不需要 session。
- 未公开接口缺少 session 时是否返回认证错误。
- 改密/重置密码是否保留 Better Auth 错误语义并映射成业务异常。
- `/account/navigation` 是否返回 menus、permissions 和 userStateVersion，且核心按钮能力仍来自业务列表接口。
