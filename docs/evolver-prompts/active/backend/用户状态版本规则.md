# 用户状态版本提示词

适用范围：用户、角色、菜单、权限变更后的状态版本刷新；前端权限/菜单缓存刷新；响应头协议。

## Prompt

你在 ShiroAdmin 中处理用户、角色、菜单、权限或会话相关改动时，要考虑用户状态版本链路。用户状态版本用于通知客户端刷新用户资料、角色、菜单和权限状态。

## 响应头协议

- `app-api` 和 `admin-api` 都使用本地响应拦截器写入用户状态版本响应头。
- `admin-api` 的 Better Auth `get-session` 成功响应也会通过插件写入 `x-user-state-version`，方便前端认证客户端同步版本。
- 两端都会返回 `x-user-state-version`。
- 客户端上送滞后版本时，服务端可能返回 `x-user-state-changed: 1`。
- 异常响应通过 `USER_STATE_HEADER_WRITER` 尝试写入版本头；失败时应不影响主响应。
- `admin-api` 的响应拦截器会跳过 `/metrics` 和 `/admin/metrics`，保持 Prometheus text/plain 响应。
- `admin-api` 正常响应路径先执行业务 handler，再写入用户状态版本头，最后包装响应体；异常响应继续由 exception filter 尝试写头。
- `apps/admin-web/src/api/index.ts` 在普通请求中携带本地 `x-user-state-version`，并从所有响应中保存最新版本。
- 非 `/account/navigation` 响应带 `x-user-state-changed: 1` 时，前端触发一次去重后的 `refreshAuthStateByVersionChange()`。
- `/account/navigation` 是刷新链内部入口，它的响应只保存版本头，不触发自身刷新，避免 `refreshAuthStateByVersionChange -> fetchPermissions -> /account/navigation` 等待自身完成。

## 核心规则

- 用户状态版本通常包含用户、角色、菜单三个维度。
- 后端在正常响应和异常响应里尽量写入用户状态版本头。
- 前端看到版本变化后刷新 Better Auth session、导航/菜单和权限状态。
- `request.session.roles` 是计算当前用户综合版本的重要输入。
- 不要为了计算版本而无需求地额外查角色表；优先使用 session 中已有角色信息。

## admin-api 关注点

- `admin-api` 使用 `AdminUserStateService`。
- Redis key 前缀为 `ver:admin:*`。
- 菜单 DB 记录名为 `admin_global`。
- 角色 DB 记录名为 `admin:role:<roleId>`；不要写入或依赖 roleName 形式的角色版本 key。
- 综合版本包含菜单版本、用户版本和当前 session 中角色版本。
- 角色按 `roleId` 去重和排序后参与哈希，减少输入顺序影响。
- 角色版本输入中保留 `roleId:version` 绑定，避免不同角色同版本值造成歧义。
- 综合版本使用 SHA256 输出，便于响应头传输和前端比较。
- 用户、角色、菜单 requiredPermissionCode、用户组、角色权限、对象例外授权、SpiceDB 数据运维直改等变更后，需要按当前模块已有方式 bump 对应版本。
- `/account/navigation` 或类似刷新接口要避免触发自身刷新循环。

当前上游模块口径：

- `SystemRolesService` 创建、更新、删除角色，分配直接用户或用户组后刷新角色版本和受影响用户版本。
- `SystemMenusService` 创建、更新、删除菜单后刷新菜单版本；菜单分配角色后刷新受影响角色版本。
- `SystemUserGroupsService` 分配成员或角色后刷新受影响用户和角色版本。
- `AuthzPermissionService.applyMatrixChanges()` 应用核心 manager relation 后刷新受影响角色版本。
- `AuthzObjectExceptionService.applyBindings()` 应用对象例外授权后刷新受影响用户和角色版本。
- `SpiceDbDataService` 对基础关系或 attribute 做运维直改并修复投影后，会刷新受影响用户、角色或菜单版本。
- `SystemUsersService` 当前接入对象例外清理、用户基础关系清理和本地投影清理；涉及用户资料、禁用、密码、会话等操作时仍需按具体代码核对是否需要补充 `bumpUserStateVersion()`。

## app-api 关注点

- `app-api` 使用 `UserStateService`。
- Redis key 前缀为 `ver:*`。
- 菜单 DB 记录名为 `global`。
- Better Auth 新用户初始化会创建 profile、绑定默认角色并刷新用户版本。
- SystemRolesService 的角色创建/更新、用户角色分配，以及 SystemMenusService 的菜单创建/更新/删除等写操作会刷新对应版本。
- `attachUserStateHeaders()` 从 session 中读取 `user.id` 和 `roles`。
- App 端菜单和动作权限读取 RBAC effective 权限与可见菜单。

## admin-web 消费

- `apps/admin-web/src/api/index.ts` 会读取并同步 `x-user-state-version`。
- 请求拦截器会把已存储版本放进 `x-user-state-version` 请求头。
- 收到 `x-user-state-changed: 1` 时，前端会触发认证状态刷新逻辑。
- `refreshAuthStateByVersionChange()` 内部已有 promise 去重，负责串行刷新 Better Auth session、`/account/navigation` 权限和动态菜单。
- Better Auth 客户端请求和普通 axios 请求对版本头的处理可能存在差异，调整时需同时核对两条链路。

## 当前风险

- 角色版本 DB 主 key 使用 `admin:role:<roleId>`。
- 用户版本只存在 Redis，Redis flush 或冷启动会让综合版本漂移。
- 涉及用户资料、禁用、密码、会话的任务，应先核对当前 Service 是否已经刷新用户版本。

## 验证建议

- 用户角色变化后，下一次普通接口响应是否带变化标记。
- 前端是否只触发一次串行刷新。
- 菜单所需权限或角色权限变化后，动态路由和按钮能力是否更新。
- 权限矩阵和对象例外授权应用后，受影响角色或用户是否能触发版本变化。
- session 缺失时是否跳过版本写入或返回认证错误，而不是产生假版本。
- 修改响应拦截器时，额外验证 `/metrics` 不被 JSON 包装。
